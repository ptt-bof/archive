■題名：実装時ドキュメントへのモジュール構造の導入と支援ツールの提案

------------------------------------------------------------------
■概要：

ソフトウェア開発で主体となるのはプログラミングだが、重要な情報を含むド
キュメントがソースコード中にコメントの形式で頻繁に記述されるため、ドキ
ュメントの役割も非常に大きい。また、読み手に誤解を与えないように正確に
ドキュメントを記述することが求められる。

理想は、ドキュメントをモジュール化して記述することである。そうすること
で、ドキュメントの保守性や再利用性が向上する。しかし、Java 用のドキュ
メント作成ツールである Javadoc は、ドキュメントをうまくモジュール化し
て記述することはできない。また、プログラムがモジュール化できていたとし
ても、それに伴ってドキュメントもモジュール化されるわけではない。一方、
ドキュメントを主体にしてプログラミングを行うための技法である文芸的プロ
グラミングでは、ドキュメントに対応してプログラムの断片が割り当てられる
ため、ドキュメントをモジュール化しやすい。しかし、文芸的プログラミング
は関数型プログラミング言語用に開発されたため、オブジェクト指向などのさ
まざまな言語パラダイムに適応させにくいという問題がある。そこで本稿は、
コメント形式で記述するドキュメントにモジュール構造を導入することを提案
する。開発者はプログラミング言語のモジュール構造に合わせてドキュメント
もモジュールごとに一度だけ記述すればよい。さらに、実装・リリース・保守
の各開発段階に対しそれぞれに最適な支援ツールを提供し、モジュールに分割
したドキュメント同士を結合することができる。支援ツールは 2つの尺度に基
づいて開発されている。モジュール化されたドキュメントの織り込み精度と、
開発者が与えなければならないヒントの量である。ヒントは、タグを用いてコ
メントの中で記述する。

実装時には、ドキュメントをモジュール化し、アスペクトによって振る舞いが
拡張されたクラスをユーザーに表示するためのドキュメントビューア Aspect-
Scope を提供する。実装時には、プログラミングが主体となるため、モジュー
ル化されたドキュメントをツールが自動で織り込みをして表示することでプロ
グラムに対する開発者の理解を助ける。そのため、開発者がヒントを与える必
要はないが、織り込み精度は低い。また、呼び出す側が拡張されたとする cal
l ポイントカットを呼び出される側が拡張されたとする execution ポイント
カットとして簡略的に解釈することで、複雑なアスペクト指向言語に対する理
解を助ける。また、実際にアスペクト指向言語のひとつである AspectJ で実
装されたソフトウェアを用い、簡略的な解釈を行っても問題がないかを検証し
ている。

ライブラリやフレームワークのリリース時には、API（Application Programming
Interface）ドキュメントを生成するためのツール CommentWeaver を提供
する。リリース時には、正確に記述された API ドキュメントをユーザーに提
供するために、織り込み精度が高くなければならない。その代わり、コメント
中にタグの形で織り込みのためのヒントを開発者が多く記述する。メソッド間
におけるドキュメントの横断・継承関係によるドキュメントの横断・アスペク
トによるドキュメントの横断に対し、それぞれモジュラーにドキュメントを記
述するためのタグを提供している。さらに、実際に広く使用されている J2SE 
 Eclipse などといったライブラリやフレームワークにおいてドキュメントの
横断があることを確認し、提案手法を用いることでドキュメントの記述量が削
減されたことを確認している。

ライブラリやフレームワークのリリース後に、ドキュメントの保守を容易にす
るためのツール Universal Aspect Oriented Programming を提案する。保守
時には、リファクタリングに対していちいち織り込みのためのタグの内容を書
き換えないのが望ましいため、タグの量は少ない方が望ましい。その代わり、
織り込み精度はやや低くなる。ドキュメントを織り込むためのタグは述語や目
的語などの文を構成する仕様を選択し、プログラムの構造は参照しないため、
リファクタリングに対してタグの内容を書き換える必要はなくなる。また、ア
スペクト指向の目的である横断的関心事の分離を、本ツールを用いることによ
りドキュメント上で実現することができるため、アスペクト指向と同等のこと
が可能となる。さらに、Java のバイトコード変換ライブラリである Javassist
を用いた場合に、実際にリファクタリングに対してタグの変更が少なく済んだ
ことを確認している。
